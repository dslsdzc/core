Core 执行模型设计书

---

一、核心原则

1.1 唯一执行模型

Core 只有一种执行语义：数据流图。

程序员编写的所有代码，经语义 IR 翻译后，构成一张有向图。图的拓扑结构决定了程序的执行方式：

图结构 执行特征
有向无环图（DAG） 按拓扑序串行执行，完全确定，天然可单步
带反馈环的静态图 循环节点按固定节奏激活，协作式调度
动态图（运行时创建/销毁节点） 并发、异步、节点生命周期动态管理

程序员不需要声明"模式"。他写的代码产生什么形状的图，程序就以对应的方式运行。

1.2 部署配置原则

物理世界的约束——内存是否可动态分配、是否多核、是否有操作系统——不属于程序的执行语义，而属于部署目标的属性。

这些属性在部署配置文件中声明，由编译器据此检查代码兼容性并生成对应代码。代码不变，部署配置可变。

1.3 渐进式能力解锁

Core 的语法特性按图结构复杂度分为三个层级：

层级 语法特性 产生的图结构 心智模型
基础 变量、函数、分支、循环（for） DAG 顺序执行
中级 loop、recv（单输入等待） 带反馈环的静态图 循环调度
高级 flow、go、yield、select 动态图 并发与通信

初学者从基础层级起步，图是 DAG，执行即文本顺序。当他们需要并发时，引入新的语法特性，图自然生长出更复杂的拓扑。不需要"切换模式"。

二、图结构决定执行

2.1 DAG：天然的教学与调试执行

```core
fn add(x: int, y: int) -> int {
    return x + y;
}

fn main() {
    a := 5;
    b := 3;
    c := add(a, b);
    println(c);
}
```

生成的图是无环的。执行器按拓扑序串行走节点。
单步调试即顺序走节点，变量值完全可预测。

这就是"教学环境"。但不需要任何标志——DAG 天然就是教学环境。

2.2 静态循环图：有节奏的循环调度

```core
fn main() {
    data := [1, 2, 3, 4, 5];
    loop {
        if data.is_empty() { break; }
        item := data.pop();
        println(item);
    }
}
```

图包含反馈边。执行器按迭代节奏执行。
仍然可串行模拟，可单步跟踪循环迭代。

2.3 动态图：并发与自适应

```core
flow worker(id: int) -> int {
    loop {
        yield compute(id);
    }
}

fn main() {
    w1 := go worker(1);
    w2 := go worker(2);
    loop {
        v1 := w1.recv();
        v2 := w2.recv();
        println(v1 + v2);
    }
}
```

图在运行时动态扩展：go 创建新节点，yield/recv 在节点间传递 token。
执行器使用协作式或多核调度，由部署配置决定。

三、资源分配：由部署配置声明

3.1 代码不规定分配策略

Core 代码只描述计算的逻辑和数据的结构，不规定内存从哪来、怎么分配。
语义 IR 保留数据的类型和边界信息（包括编译期已知的上限），但不固定分配策略。

3.2 部署配置描述目标平台能力

```toml
[deploy.mcu]
target = "cortex-m4"
allocation = "static"          # 禁止动态分配
max_memory = "256KB"

[deploy.server]
target = "x86-64-linux"
allocation = "dynamic"         # 允许动态分配
```

编译时：

· 目标为 mcu：编译器检查所有可变大小数据结构是否已有编译期确定的上限。若无法确定，拒绝编译并给出明确错误。
· 目标为 server：编译器生成使用目标平台分配器的代码，允许运行时决定大小。

程序员仍然只写一份代码。如果这份代码能在静态分配世界运行，它必须足够保守；如果只能在动态分配世界运行，它就对部署环境有要求。

3.3 资源配额：运行时可选策略

资源配额（CPU 时间片、内存上限、I/O 速率）不属于程序逻辑，属于运营策略。
在部署配置中可选声明：

```toml
[deploy.server.quotas.flow.worker]
max_memory = "10MB"
cpu_budget = "5ms"
```

不声明即无限制。代码不变。

---

四、时间语义：由部署上下文决定

程序代码中不区分"零时间""物理实时""仿真时间"。
时间是一个可选的外部输入源：

· 裸机或实时 OS：硬件时钟提供物理时间，调度器可据此满足 deadline。
· 普通 OS：系统时钟提供逻辑时间，保证事件偏序。
· 仿真环境：仿真器提供可控时间，可加速、暂停、回放。

部署配置决定时间源，而非代码决定。

---

五、执行策略：编译期自动选择

部署环境 编译器自动选择
单核嵌入式 串行 DAG 或静态循环调度，零运行时开销
多核服务器 多核并行调度，节点映射到线程池
浏览器/Wasm 协作式事件循环
分布式集群 远程 go，边映射为网络消息

程序员不指定"调度策略"。编译器根据目标环境自动生成最优方案。
语义 IR 不变。

六、静态与动态图，不是模式切换

Core 区分静态图和动态图，但这不是模式——它们是图的两个形态：

6.1 静态图

编译期已知所有节点和边。
类型：DAG、带循环的静态图。
执行：拓扑序、循环调度。
用途：计算、控制逻辑、无动态创建的流。

6.2 动态图

运行时创建/销毁节点、重连边。
类型：带有 go、动态 select 的图。
执行：动态节点生命周期管理。
用途：长期服务、自适应系统、热升级。

初学者从静态 DAG 起步，逐步解锁动态图特性。这不需要"切换模式"，只是使用更多的语言能力。

七、代码一次编写，处处自适应

```core
fn compute(x: int) -> int { x * 2 }

fn main() {
    result := compute(42);
    println(result);
}
```

这段代码：

· 在教学环境 → DAG，拓扑序解释执行，可单步、可回退
· 在桌面 → AOT 编译，直接原生执行
· 在嵌入式中 → 静态分配，零运行时
· 在分布式集群 → 可远程部署 compute

代码不改。图的语义相同。部署配置不同。

---

八、与传统"多模式语言"的区别

传统做法 Core
代码中声明模式（#[mode(release)]） 代码无模式概念
不同模式影响语义或行为 图语义唯一
切换模式可能引入 bug 部署配置不影响程序正确性
模式数量膨胀 无模式分类

Core 只有一种执行语义：图。
图结构由代码决定，物理约束由部署配置声明，执行策略由编译器自动选择。

---

九、结论

Core 的执行模型可以总结为一句话：

程序员画图，编译器认图，部署配置描述世界。图上没有"模式"，只有拓扑。拓扑简单则执行简单，拓扑复杂则执行复杂。一切自然而然地发生，不需要人为命名任何一种"模式"。
